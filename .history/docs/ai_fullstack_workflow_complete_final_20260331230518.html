AI 全栈开发工作流 · 自动化 vs 人工执行明细

每个工具标注执行模式 + 人工检查点 + 自动化设计
Java + React Native + Next.js · 阿里云 + GitLab

自动 Agent 自主执行，无需人工介入 门禁 自动执行但有强制检查点，不过不放行 人工 需要人类确认、签字或决策
1

需求解析 parse-requirements

输入：需求文档 + 设计稿 + 历史 specs/输出：proposal.md + specs/ + 需求边界
自动 Superpowers brainstorming — 六步需求分析
自动化设计：session 启动后 agent 检测到新任务，自动加载 brainstorming skill，进入反问流程。无需手动输入 /brainstorm。
  • Step 1-2（问题定义/用户场景）：agent 自动生成追问，展示在终端等待回答
  • Step 3-4（范围边界/分段展示）：agent 自动拆分设计为可消化的段落
  • Step 5-6（输出文档/签字确认）：agent 自动写入项目目录
人工检查点：每段设计展示后等待你确认（回复 yes / 提修改意见）。agent 不会自动跳过——这是 Superpowers 的 HARD-GATE，未确认则流程阻塞。
自动 Pencil MCP — 读取设计稿
自动化设计：agent 检测到 .pen 文件在项目中，自动调用 Pencil MCP 读取页面结构、组件列表、交互流，转为文本描述写入 specs/ 的 UI 部分。
无需人工：提取结果作为后续 UI 验收基线，人工在 Phase 1 结束时统一审阅。
自动人工 OpenSpec /opsx:propose — 需求固化
自动化设计：brainstorming 完成后，agent 自动调用 /opsx:propose 生成 proposal.md + specs/。
  • proposal.md（背景/目标/范围/排除项）：agent 根据澄清结果自动生成
  • specs/（Given/When/Then）：agent 逐条从设计摘要中提取行为场景
  • RFC 2119 关键词（MUST/SHOULD/MAY）：agent 自动标注
人工检查点：proposal.md 和 specs/ 生成后，需要你逐条审阅确认。重点检查：scope 边界是否正确、Given/When/Then 是否覆盖所有场景、需求强度标注是否合理。确认后 agent 才进入 Phase 2。
2

方案设计 design-architecture

输入：proposal.md + specs/ + 现有代码库输出：design.md + tasks.md + API 契约 + 数据模型
自动人工 Superpowers designing-before-coding
自动化设计：agent 读取 specs/ 和 proposal.md，自动生成 design.md + tasks.md。
  • 技术选型 + 架构图 + 认证策略 + 数据模型：全自动生成
  • tasks.md 拆解：每个任务精确到文件路径/代码/验证步骤：全自动
  • 禁止 TBD/占位符——skill 内置检查器自动拦截模糊描述
人工检查点：design.md 生成后展示给你。重点审阅：架构是否合理、三端契约是否一致、认证策略是否安全。这是"最后一次人工把关"——确认后 agent 将自主执行数小时。
自动 Subagent 交叉审查 (Opus)
自动化设计：主 agent 自动启动第二个 Claude (Opus) 作为 Staff Engineer 审查 design.md。
  • 审查维度：安全漏洞/性能瓶颈/扩展性/三端一致性
  • 输出 review comments 并标注严重度
  • 主 agent 根据 comments 自动修改 design.md
  • 循环直到审查 subagent 输出 "approve"
全自动：两个 agent 之间的审查-修改循环无需人工介入。
3

并行编码 generate-code

输入：design.md + tasks.md + API 契约 + 各端 CLAUDE.md输出：三端代码 + 单元测试 + 3 个独立分支
自动 Superpowers subagent-driven-development
自动化设计：人工确认 design.md 后，agent 进入全自主执行模式（可持续数小时）：
  • 自动创建 3 个 git worktree（各端独立分支 + 目录）
  • 按 tasks.md 顺序，为每个任务启动全新 subagent
  • 每个 subagent 自动加载对应端的 CLAUDE.md + skill
  • subagent 内部自动执行 TDD + 两阶段 review
  • 遇到阻塞才停止，否则持续推进
Agent AJava · 自动读 java-service skill
Agent BRN · 自动读 rn-mobile skill
Agent CNext.js · 自动读 nextjs-web skill
全自动：这是 agent 自主工作最长的阶段（1-4h）。你可以去做别的事。
自动 Superpowers TDD — 每个 subagent 内部
自动化设计：skill 强制注入每个 subagent 的执行流：
  • 红：subagent 自动写一个失败测试
  • 绿：自动写最少代码让测试通过
  • 重构：自动清理代码并确认测试仍绿
  • 铁律执行：skill 检测到先写了代码没测试 → 自动删掉代码重来
全自动：TDD 循环完全在 subagent 内部闭环。
门禁 Hint Hook — 非阻塞规范提示
自动化设计：PostToolUse hook 在 agent 每次写文件后自动触发，检查是否偏离规范，偏离时注入提示但不阻塞。日志记录供 Phase 7 分析。
全自动：后台守护进程，人和 agent 都不需要额外操作。
4

测试验证 run-verification（10 项）

输入：三端代码 + 测试 + specs/ + 设计稿 + DB migration输出：10 项全绿 + 报告 + 可 commit 代码
自动1. 功能用例
自动2. 单元测试
自动3. E2E
自动4. API 契约
自动5. 安全扫描
自动6. 性能基准
自动7. DB 迁移
自动8. UI 验收
自动9. 视觉回归
门禁10. 回修循环
自动 Superpowers verification-before-completion
自动化设计：agent 声称"完成"时 skill 自动拦截，必须实际跑命令并展示输出。
全自动：1-9 项检查全部由 agent 自主执行。失败时自动进入修复循环。
门禁 Block-at-submit Hook — commit 强制门禁
自动化设计：PreToolUse hook 拦截 git commit：
  • 检查 pass 标记文件 → 存在放行 / 不存在阻止
  • 阻止后 agent 自动定位→修复→重跑→再尝试 commit
全自动：回修循环完全在 agent 内闭环。hook 是确定性脚本，不依赖 agent 自觉。
自动 Playwright skill + Pencil MCP
Playwright：E2E 跨浏览器 + 截图 pixel diff 视觉回归。Pencil MCP：设计稿标注 vs DOM 对比 UI 验收。
人工可选：量化指标自动对比，"感觉对不对"建议看截图（附在 MR 中）。
5

代码审查 code-review

输入：全绿代码 + design.md + specs/输出：审查通过的 MR + 安全报告
自动 Superpowers requesting-code-review — 两阶段
第一阶段 spec 合规 + 第二阶段 code quality，发现问题自动修复并循环。
人工检查点：merge / keep / discard 最终决定权在你。
自动 security-reviewer subagent (Opus)
独立 context 安全审查：注入/认证/敏感数据/硬编码密钥。Critical 级 → 主 agent 自动修复。
全自动：审计报告附在 MR 中供浏览。
自动人工 GitLab MCP — MR 管理
自动创建 MR + 附 review/安全报告/截图。
人工检查点：最终 approve + merge 由你点击。审阅 MR 描述 + 安全报告 + 截图，~5 min。
6

自动化部署 deploy

输入：已 merge MR + Dockerfile + K8s manifest输出：三端上线 + specs/ 归档 + 部署日志
自动 GitLab CI Pipeline
MR 合并后自动：Docker build → push ACR → ACK 滚动更新 → 健康检查 → SLB 切流。
全自动：merge 到上线全程 CI/CD。失败自动通知。
自动 OpenSpec archive + worktree 清理
delta specs 合并到主 specs/。Superpowers finishing-a-development-branch 清理 worktree。
人工可选：worktree 清理前 agent 确认，一般直接同意。
7

自学习沉淀飞轮

输入：CI 日志 + review 记录 + hint hook 日志输出：新 SKILL.md + 更新 CLAUDE.md + 累积 specs/
自动人工 错误模式分析
自动分析 CI 日志提取重复错误（3+ 次触发 skill 沉淀建议）。
人工检查点：确认"这值得固化为 skill"，避免 skill 膨胀。
自动 Superpowers writing-skills (TDD)
确认后 agent 用 TDD 写 skill：红（观察犯错）→绿（写 SKILL.md）→重构（压力测试）。
全自动：skill 写完后下次会话自动生效。
门禁 硬编码审计 Hook
PostToolUse hook 检测 .claude/skills/ 和 openspec/ 写入中的非 {{VAR}} 格式 → 阻止。
全自动门禁：不依赖 agent 自觉，hook 硬拦截。
人工介入点总览（整个大需求流程仅 4 处）
人工 Phase 1 结束：审阅 proposal.md + specs/，确认需求边界
人工 Phase 2 中间：审阅 design.md，确认架构方案（最后一次人工把关）
人工 Phase 5 结束：审阅 MR + 安全报告 + 截图，点击 approve + merge
人工 Phase 7 触发：确认"是否值得沉淀"，避免 skill 膨胀
其余全部环节（编码 / TDD / 测试 10 项 / 回修循环 / 安全审查 / 部署 / 归档）均由 agent + hook 全自动执行。
